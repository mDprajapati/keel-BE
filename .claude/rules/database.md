# Rule: Database & multi-tenancy

Source: `Keel-MVP-Timeline-backend` (Data models & migrations), v3 §9, §13.

## Tenancy

- **Every domain row carries `workspace_id`** — the hard multi-tenant boundary. Derive it from the authenticated principal, never from request input.
- Every retrieval/list query filters by `workspace_id`. Every Qdrant search passes a mandatory `workspace_id` payload filter. Every chunk row carries `workspace_id`.

## Schema conventions

- Every table: `id` (UUID pk), `created_at`, `updated_at` (BOTH timestamps, always).
- SQLAlchemy 2.0 async (`AsyncSession`). Services own the transaction (commit/rollback); Celery tasks open their own session.
- Migrations are **relational only** (Alembic) — no vector/graph DDL (Qdrant + Neo4j own theirs). Always review autogenerated migrations before commit.
- Enums match `docs/data-model.md` exactly (e.g. `role ∈ {admin, standard}`, `ingestion_status` full granular set, `api_key_scope ∈ {read_only, read_write}`, `connector_status ∈ {connected, disconnected, coming_soon}`).

## Skeleton columns (present, not enforced)

`documents.content_hash`, `documents.duplicate_of` (SHA-256 dedup deferred §9.2); `users.is_verified` (email verify deferred §6.2); `users.mfa_enabled` (MFA deferred §6.3). Build the column; do not gate behaviour on it.

## Response shaping

Never return ORM objects from a router — map to a Pydantic schema in `app/schemas/`. The document response field names match `KeelDocument` (`name`, `uploaded_by`, `uploaded_at`, `chunk_count`, …).
